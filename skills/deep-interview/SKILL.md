# Deep Interview - ソクラテス式要件掘り下げ

discovery-council の前段。要件が曖昧なときに使う。**まだ plan が無い**段階で要件を引き出すのが役割。逆に**既にある plan/design を詰問**したいなら `/flywheel:grill`（こちらは掘り下げでなく stress-test）。

> **対話前提スキル**: 1 問ずつの掘り下げが本質なので、AskUserQuestion が使えない環境では「要件が曖昧です。対話モードで /deep-interview を実行してください」と案内して終了する。

## 入力

`$ARGUMENTS`（ユーザーの曖昧なアイデア・要望）

## 原則

- **1回に1つの質問**。バッチで聞かない
- 前の回答を踏まえて次の質問を決める
- 選択肢を提示する（答えやすくする）
- 「なぜ？」「たとえば？」「もし○○なら？」の形で掘り下げる
- **止めるのは人間**。モデルから「もう十分」と切り上げない（聞かれるのは負担ではない＝ under-ask が失敗）。曖昧軸が残る限り1問ずつ続ける

## 5軸の曖昧性評価

各質問の前に、以下の5軸で内部評価する（ユーザーには見せない）:

| 軸 | 問い |
|----|------|
| **WHO** | 誰のための機能か？ユーザーは誰か？ |
| **WHAT** | 何を実現するか？具体的な振る舞いは？ |
| **WHY** | なぜ必要か？現状の課題は何か？ |
| **SCOPE** | 何を含み、何を含まないか？ |
| **CONSTRAINT** | 技術的制約、期限、依存は？ |
| **DONE** | 何が満たされたら完了か？それを機械的に検証するコマンドは？（flywheel の eval_cmd の種。design.md の「## 完了条件（eval）」に育つ） |

DONE 軸は省略しない——**自分で完了条件の案を設計して提示し、ユーザーには承認/修正だけさせる**（ゼロから聞かない）。既存のテスト構成を Glob/Grep で見て「`pytest tests/test_X.py` が通ること、でよいですか？」の形まで落とす。

## Step 1: 初期分析

ユーザー入力を読み、5軸それぞれを [明確 / やや曖昧 / 曖昧] で評価する。既存コードベースがある場合は Glob/Grep/Read で関連コードを確認し、質問の精度を上げる。

## Step 2: インタビューループ

最も曖昧な軸から順に1問ずつ質問する。

```yaml
AskUserQuestion:
  question: "<最も曖昧な軸に対する具体的な質問>"
  options:
    - "<選択肢A>"
    - "<選択肢B>"
    - "<選択肢C>"
    - "自由に記述"
```

**質問の形:** 「なぜ？」「たとえば？」「もし○○なら？」で掘り下げ、必ず3-4個の選択肢 + 自由記述を提示する。

**ループ終了条件（止めるのは人間）:** モデルは終了を自己判定しない。止める直前に、まだ詰めていない **未決の判断**（残った曖昧軸）を提示して人間に stop/continue を聞く。**この提示は prose でなく AskUserQuestion で出す**: 残り判断の枝を選択肢に（効く上位3 + 4つ目「握れた・進めて」・single-select）。枝を選べばその軸を詰めて再 checkpoint、「握れた・進めて」で stop。残り軸が4個超なら質問文に「他に N 個」と添える。人間が「握れた・進めて」と言うまで、曖昧軸が残る限り続ける（informed stop）。

**回答への対応:**
- 「分からない」→ 仮定を提案して確認を取る（「○○と仮定してよいですか？」）
- 抽象的な回答 → 具体例を求める（「たとえばどんなケースですか？」）
- 矛盾する回答 → 優先順位を聞く（「○○と△△は両立しにくいですが、どちらを優先しますか？」）

## Step 3: 要件サマリーの確認

掘り下げた結果を5軸で構造化し、最終確認する（OK / 修正あり）。

## Step 4: discovery-council へハンドオフ

確認が取れたら、整理した要件をそのまま discovery-council に渡す:

```
Skill: discovery-council
Args: <整理された要件サマリー>
```

## Gotchas

- **早すぎる打ち切り（self-graded stop）**: モデルが「もう十分」と自己判定して止めるのが失敗モード。**止めるのは人間**で、モデルは **未決の判断**（残った曖昧軸）を提示してから人間に委ねる。ただし *事実*（コード/既存実装に答えがある）は self-answer して無駄質問を減らす——止まらないのは *判断* が残る間だけ（無限質問にしない）
- **既にある plan を掘り下げ直さない**: 完成した plan/design を持っているなら、掘り下げでなく `/flywheel:grill`（詰問）を使う。deep-interview は「ゼロから要件を引き出す」用途
- **初期分析を飛ばすと無駄質問**: Step 1 で5軸評価＋既存コード確認をせずに質問を始めると、コードを見れば分かることを聞いてループが伸びる
- **「分からない」を放置しない**: 仮定を提案して確認を取る。抽象回答は具体例を、矛盾回答は優先順位を聞く

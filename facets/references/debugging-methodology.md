# Debugging Methodology Reference

> このファイルは `agents/debugger.md` から段階的開示（Progressive Disclosure）で分離されたリファレンスです。
> エージェント実行時に `Read` して適用してください。

## 4つのフェーズ

各フェーズを完了してから次に進むこと。

### Phase 1: 根本原因の調査

**修正を試みる前に必ず実行:**

1. **エラーメッセージを正確に読む**
   - スタックトレースを全て読む
   - 行番号、ファイルパス、エラーコードを記録
   - 飛ばし読みしない

2. **再現を確認**
   - 確実にトリガーできるか？
   - 正確な手順は？
   - 毎回発生するか？
   - 再現できない → データ収集。推測しない。

3. **最近の変更を確認**
   - 何が変わったか？
   - diff、直近のコミット
   - 依存関係の更新、設定変更

4. **データフローを追跡**
   - 不正な値の発生元はどこか？
   - 誰がその値を渡したか？
   - 源流まで遡る
   - 症状ではなく源流を修正

### Phase 2: パターン分析

1. **動作する類似コードを見つける**
   - 同じコードベース内の動作例
   - 壊れたコードと何が違うか？

2. **差分を特定**
   - 動作するものと壊れたものの全ての差分をリスト
   - 「関係ないだろう」と思っても含める

### Phase 3: 仮説と検証

1. **単一の仮説を立てる**
   - 「X が根本原因だと考える。理由は Y」
   - 具体的に。曖昧にしない。

2. **最小限の変更でテスト**
   - 一度に1つの変数だけ変える
   - 複数の修正を同時にしない

3. **検証してから続行**
   - 成功 → Phase 4 へ
   - 失敗 → 新しい仮説を立てる
   - 失敗の上に追加修正を重ねない

### Phase 4: 修正

1. **失敗するテストケースを作成**
   - 最小限の再現テスト
   - テストがあることを確認してから修正

2. **単一の修正を実装**
   - 根本原因に対処する1つの変更
   - 「ついでに」の改善をしない

3. **検証**
   - テストが通るか？
   - 他のテストを壊していないか？

4. **3回以上修正に失敗した場合**
   - **停止**
   - アーキテクチャの問題を疑う
   - AskUserQuestion でユーザーに相談

## Red Flags — こう思ったら止まれ

- 「とりあえずこれを変えてみよう」
- 「複数まとめて変更して、テスト実行」
- 「テスト省略、手動で確認」
- 「たぶん X だろう、直そう」
- 「完全には理解してないけど動くかも」
- 「もう1回だけ修正を試す」（既に2回以上失敗）

**全て Phase 1 に戻るサイン。**

## 言い訳 vs 現実

| 言い訳 | 現実 |
|--------|------|
| 「シンプルだからプロセス不要」 | シンプルなバグにも根本原因がある |
| 「緊急、時間がない」 | 体系的デバッグの方が推測より速い |
| 「まずこれを試して、ダメなら調査」 | 最初の修正がパターンを決める。最初から正しくやれ |
| 「テスト後で書く」 | テストなき修正は定着しない |
| 「もう1回だけ」（2回失敗後） | 3回失敗 = アーキテクチャの問題。パターンを疑え |

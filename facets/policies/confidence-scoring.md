# Confidence Scoring - 共通ポリシー

**確信度を付けて全件報告し、集約側でフィルタする**ための共通スコアリング基準（coverage-first）。

## スコアリング基準（0-100）

| スコア | 確信度 | 例 |
|--------|--------|------|
| 90-100 | 確実な問題 | SQL injection、hardcoded secrets、明らかな NPE、無限ループ |
| 80-89 | 高い確信 | `shell=True`、N+1、エラーハンドリング不備、未使用インポート |
| 60-79 | 中程度 | 「もっと良い書き方があるかも」、軽い設計上の懸念 |
| 0-59 | 低い確信 | コンテキスト不足で判断保留、プロジェクト固有規約の可能性 |

## 報告ポリシー: Coverage-first（重要）

**検出した issue は `confidence` と `severity` を付与して全件報告する。**
agent 側で finding 時にフィルタリング・閾値カットを行わない。集約・ランキング・降格は Council 側 / 上位スキル側で行う。

### 理由（4.7+ / 4.8 リテラル解釈トラップ）

Claude Opus 4.7 から observed の挙動として `"Confidence 80+ のみ報告"` `"only critical"` `"be conservative"` のような指示を文字通り守る傾向が強い（v2.1.154 でデフォルトとなった Opus 4.8 でも同系列の傾向は継続の可能性、確証なし）。**bug を発見しているのに「閾値以下」と判定して silent drop する**事故が観測されている（Anthropic 公式 prompting best practices で名指しの注意あり）。

→ agent には「全件挙げ + confidence/severity 付与」を指示し、フィルタは集約側で持つ。閾値は集約側に集中させる方が後から調整が効く。

## 出力フォーマット

各 finding に必ず付与する:

| フィールド | 内容 |
|---|---|
| `confidence` | 0-100 の整数（あなたの判断） |
| `severity` | `critical` / `high` / `medium` / `low`（impact ベース） |
| `category` | 何の観点で見つけたか（コード品質 / セキュリティ / 計画整合性 等） |
| `file:line` | ファイルパスと行番号 |
| `issue` | 問題の要約 |
| `fix` | 解決策（具体的に） |

## 集約側の降格ルール（既定値、上位スキルが運用）

quality-gate / Review Council など集約側は、以下の既定マトリクスで降格・分類する:

| confidence | severity | 表記 | ユーザー提示 |
|---|---|---|---|
| 90+ | critical / high | 🔴 Critical（必須修正） | 必ず提示 |
| 80-89 | high / medium | 🟡 Warning（推奨修正） | 提示 |
| 60-79 | medium / low | ℹ️ Note（参考） | 詳細レポートに含める / サマリ集計外 |
| < 60 | - | 📦 Archive | 履歴用、デフォルト非表示 |

閾値はプロジェクト都合で調整可能。集約側で「Critical なし」を判定する際は **🔴 のみ**でカウントする（🟡 や ℹ️ は通過）。

## Anti-Slop Bias

「正しく見える」コードにこそ負債を疑う。LLM 生成コードは表面的にクリーンでもハッピーパスしかカバーしていない・過度に抽象化されている・エッジケースを silent fail している可能性がある。以下のパターンは confidence を **+10 して評価**する:

- 汎用的すぎる命名（`handleData`, `processInput`, `utils`）
- テストがモックだらけで実際の I/O を検証していない
- エラーを握りつぶしている（空の catch, `|| true` の多用）
- ハッピーパスのみのテスト（境界値・異常系なし）

## 共通原則

1. **Coverage-first**: 検出時にフィルタしない。confidence/severity を付けて全件報告する
2. **具体的に**: ファイル名と行番号を必ず明記
3. **建設的に**: 問題だけでなく解決策も提示
4. **降格は集約側**: 閾値判断を agent に委ねない。上位スキルで一元管理する

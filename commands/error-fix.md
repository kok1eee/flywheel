エラー修正モードを開始します。対象: `$ARGUMENTS`（空なら「どのエラー/症状ですか？」と1問だけ確認してから進む）。

このモードは flywheel の状態機械（goal / eval / backlog）には**乗りません**（stateless）。修正が goal 級の作りごと（設計判断を伴う機能実装）に化けたら、このモードを閉じて `/flywheel:start` へ切り替えてください。

## 1. 根因特定（推測修正の禁止）

- **自明なバグ**（typo・import 漏れ・明白な設定ミスで、根因が一目瞭然）: 直接修正へ進んでよい
- **非自明**（再現条件不明・仮説が複数・前回の修正が効かなかった・時間に追われて「明らかな修正」が見える気がする）: **Agent(flywheel:debugger) に委譲**し、根本原因の特定を待ってから修正する。根因調査なしの修正提案は禁止

## 2. 修正 + 検証

`Skill: flywheel:verification` の Iron Law に従う: **検証コマンドを今実行し、出力を最後まで読み、証拠を添えてから**「修正済み」を宣言する。最低ラインは「元の症状が再現しないことを示す実行結果」。

## 3. 教訓ゲート（このモードの本体・スキップ禁止）

修正の完了宣言のあと、閉じる前に必ず判定する。

### a. 不変性フィルタ

**「このミスは、モデルが賢くなっても踏むか？」**

- **No**（今のモデル/環境の一過性・単純ミス）→ 何も残さず終了してよい。**大半はここで終わる**（typo に教訓は要らない。ドクトリン層を薄めるものは載せない）
- **Yes**（外部世界の制約・組織ルール・運用環境の罠に根がある）→ b へ

### b. 2回ルール

auto-memory（MEMORY.md とその配下）を検索し、**同型の過去事例**を探す:

- **初回** → memory に feedback として記録する（無確認・自動。根因 / **Why** / **How to apply** を含め、関連 memory へ `[[name]]` リンクを張る）
- **2回目以降**、または**決定的に判定できる**（コマンドパターン・ファイルパス等で機械検査可能）→ rule / hook への**昇格を提案**する。ただし**恒久層は無断で書かない**: AskUserQuestion で1問（昇格する / memory のまま）を確認してから書く

### c. 記帳先（SoT はユーザー側）

ユーザーのドクトリン索引 `~/.claude/doctrine.md` が**存在すれば Read し、そのタグ体系・昇格フロー・記帳規約に従う**。このコマンドは**タグ体系を定義しない**（複製もしない）。doctrine.md が存在しなければ b の memory 記録までで終える。

### d. hook 昇格の場合

**発火テスト**（サンプル JSON を stdin に流して deny / ask / 沈黙を検証）を必ず同梱してから完了とする。テスト無しの hook 昇格は「登録されたが一度も発火確認されていない死に hook」を生む。

## 出力

修正報告の末尾に教訓ゲートの判定結果を1行添える:

- `教訓ゲート: 一過性（記録なし）`
- `教訓ゲート: memory 記録（<name>）`
- `教訓ゲート: rule/hook 昇格提案 → <ユーザー判断と結果>`
